📘 פרויקט AWS מלא – מדריך עבודה עם הסברים (Client + Server)
________________________________________
🎯 מטרת העבודה
בתרגיל זה נבנה מערכת אמיתית בענן הכוללת:
•	אתר (Frontend) שמוצג למשתמש
•	שרת (Backend) שמבצע לוגיקה
•	תהליך אוטומטי שמעלה את הקוד לענן
📌 המטרה אינה רק “שזה יעבוד”, אלא להבין:
•	איפה כל חלק במערכת נמצא
•	איך הם מתקשרים אחד עם השני
•	מה תפקיד כל שירות ב־AWS
________________________________________
🧩 מבט כללי על המערכת
לפני שמתחילים — חשוב להבין את התמונה:
•	ה־Client (React) → נשמר ב־S3
•	ה־Server (API) → רץ ב־ECS
•	הקוד → מגיע מ־GitHub
•	ה־Docker Image → נשמר ב־ECR
👉 כלומר:
Client → S3
Server → ECS
Images → ECR
Automation → GitHub Actions

🟢 שלב 2 – העלאה ל־GitHub אישי
למה זה חשוב?
GitHub משמש כנקודת התחלה לכל התהליך.
כל שינוי בקוד יפעיל את תהליך ההעלאה לענן.
מה עושים:
1.	להיכנס ל־GitHub
2.	ללחוץ על New repository
3.	לתת שם:
o	client-project
o	api-project
4.	להעלות כל פרויקט בנפרד
📌 חשוב:
יש לשמור על מבנה הקבצים המקורי.
________________________________________
🟢 שלב 3 – יצירת משתמש IAM- בAWS
למה לא לעבוד עם root?
משתמש root הוא מסוכן — יש לו גישה מלאה.
בפרויקטים אמיתיים תמיד עובדים עם משתמש מוגבל.
מה עושים:
1.	להיכנס ל־IAM
2.	Users → Create user
הגדרות:
•	לבחור שם משתמש
•	לסמן Programmatic access
הרשאות:
•	לבחור Attach policies directly
•	לבחור: AdministratorAccess
📌 הסבר:
בתרגיל נותנים הרשאה מלאה כדי לא להיתקע. במציאות נותנים רק מה שצריך.
________________________________________
יצירת Access Keys
1.	להיכנס למשתמש
2.	Security credentials
3.	Create access key
📌 יש לשמור:
•	Access Key ID
•	Secret Access Key
👉 אלו הסיסמאות ש־GitHub ישתמש בהן כדי לעבוד מול AWS.
________________________________________
🟢 שלב 4 – יצירת S3 (Frontend)
מה זה S3?
שירות אחסון קבצים.
ניתן להשתמש בו גם כדי להציג אתר סטטי.
מה עושים:
1.	S3 → Create bucket
2.	לבחור שם ייחודי
⚠️ חשוב מאוד:
לבטל:
Block all public access
📌 הסבר:
בלי זה — האתר לא יהיה נגיש מהאינטרנט.
________________________________________

1.	להיכנס ל־Bucket
2.	Properties
3.	Static website hosting → Enable

•	index.html

👉 (עם שם ה־bucket שלך)

🟢 שלב 7 – חיבור GitHub ל־AWS (Client)
למה צריך Secrets?
אסור לשים סיסמאות בקוד.
GitHub מאפשר לשמור מידע רגיש בצורה מאובטחת.
מה עושים:
1.	להיכנס ל־Repository
2.	Settings
3.	Secrets and variables → Actions
4.	New secret
להוסיף:
•	AWS_ACCESS_KEY_ID
•	AWS_SECRET_ACCESS_KEY
•	AWS_REGION
•	S3_BUCKET
________________________________________
🟢 שלב 8 – Deploy ראשון (Client)

מה קורה מאחורי הקלעים?

🟢 שלב 9 – יצירת ECR
מה זה ECR?
מקום שבו שומרים Docker Images.
📌 חשוב להבין:
ECR לא מריץ קוד — רק שומר אותו.
מה עושים:
1.	ECR → Create repository
2.	לבחור Private
3.	לתת שם
________________________________________
🟢 שלב 10 – ECS (הרצת השרת)
מה זה ECS?
שירות שמריץ Containers (Docker)
📌 הסבר פשוט:
במקום לנהל שרת — נותנים ל־AWS להריץ את האפליקציה.
________________________________________
יצירת Cluster
1.	ECS → Create cluster
2.	לבחור Fargate
📌 Fargate = לא צריך לנהל שרתים
________________________________________
יצירת Task Definition
מה זה?
הגדרה של איך להריץ את האפליקציה.
כולל:
•	איזה image
•	איזה פורט
•	כמה זיכרון
________________________________________
🟢 שלב 11 – Service
מה זה Service?
דואג שהאפליקציה תמיד תהיה רצה.
📌 אם היא נופלת — הוא מעלה אותה מחדש.
________________________________________
⚠️ הגדרות רשת (קריטי להבין!)
Public IP
אם לא מפעילים — לא ניתן לגשת לאפליקציה.
Subnet ציבורי
מאפשר גישה מהאינטרנט.
________________________________________
⚠️ Security Group
זהו Firewall.
אם לא פותחים פורט → אי אפשר לגשת.
יש להוסיף:
•	Port 3000
•	Source: Anywhere
